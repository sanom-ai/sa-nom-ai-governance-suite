# Issue Drafts - v0.4.1

Use these issue drafts to deliver the `v0.4.1` PTAG polish-and-hardening milestone.

## 1. Add A Short PTAG Quick Start

**Goal**
Give new readers a very short path that explains how to start reading and using PTAG in the repository.

**Scope ideas**
- add a short-start PTAG doc
- point readers to the minimum useful example set
- link the quick-start path from README or docs index

**Success signal**
A new reader can understand where to begin with PTAG in a few minutes.

## 2. Expand PTAG Public Examples

**Goal**
Add more examples so PTAG is easier to study from simple syntax through enterprise-shaped usage.

**Scope ideas**
- add small examples focused on one block at a time
- add a more complete multi-block example
- explain what each example is trying to demonstrate

**Success signal**
Readers can move from PTAG concept to usable example patterns without guessing.

## 3. Add PTAG FAQ And Comparison Notes

**Goal**
Answer the most likely PTAG questions before maintainers have to answer them repeatedly by hand.

**Scope ideas**
- explain what PTAG is for
- explain what PTAG is not trying to replace
- compare PTAG at a high level with adjacent policy or orchestration tools where helpful

**Success signal**
Readers can evaluate PTAG faster and with less confusion.

## 4. Harden PTAG Syntax And Semantic Tests

**Goal**
Improve confidence that PTAG docs and implementation stay aligned as the public surface grows.

**Scope ideas**
- add tests for header requirements
- add tests for unsupported block types
- add tests for semantic role/authority mismatches and policy-coverage-gap behavior

**Success signal**
PTAG changes become easier to trust and review.

## 5. Close v0.4.1 With PTAG Adoption Polish

**Goal**
Finish the milestone with visible proof that PTAG became easier to learn and safer to evolve after its public opening.

**Scope ideas**
- release notes for `v0.4.1`
- short summary of PTAG onboarding improvements
- note any remaining gaps reserved for later tooling phases

**Success signal**
`v0.4.1` feels like a strong follow-through release after `v0.4.0`, not a vague add-on.
