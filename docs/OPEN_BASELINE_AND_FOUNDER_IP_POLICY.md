# Open Baseline And Founder IP Policy

This document explains how SA-NOM should separate:
- the real open-source baseline that already exists in this repository
- future founder-owned innovations that may remain outside the public repository
- the commercial layer around rollout, support, and high-trust delivery

The goal is to keep the public baseline honest while protecting room for future proprietary innovation.

## Core Rule

SA-NOM should continue to treat the public repository as a real product baseline, not a teaser edition.
Future premium innovation should come from new advanced layers, new private modules, founder-owned calibration logic, or quote-specific commercial assets that are intentionally developed outside the public baseline.

That means:
- what is already public in the repository stays part of the public baseline
- future advanced innovation does not need to be added to the public repository by default
- commercial value may come from both delivery work and future founder-owned innovation layers

## What The Public Baseline Already Means

Through the `v0.7.9` line, the public repository already includes a real governed runtime baseline such as:
- PTAG governance language
- Role Private Studio
- Human Ask
- command surface and Control Room baseline
- cases, documents, AI actions, and work-routing surfaces
- master data, search, and assignment baseline
- audit and evidence posture
- private-first deployment and runtime confidence tooling

This baseline should remain usable for:
- evaluation
- self-managed learning
- internal experimentation
- self-managed private deployment
- community improvement inside the public scope

## What May Become Founder-Owned From Now On

Starting from new work that has not yet been released into the repository, SA-NOM may develop future innovation outside the public baseline.

This can include areas such as:
- advanced PT-OSS layers beyond the public-safe foundation
- premium calibration logic and scoring profiles
- sector-specific intelligence packs
- sovereign or regulated environment packs
- proprietary private registry content
- premium evaluation bundles or benchmark suites
- founder-owned workflow packs, decision profiles, or role libraries
- advanced commercial-only operational modules that are intentionally kept outside the public repository

The key principle is simple:
future founder-owned work must be clearly separated before release, not retroactively reclaimed after public release.

## PT-OSS Boundary

SA-NOM should use a layered PT-OSS posture:

### PT-OSS Foundation

This is the public-safe foundation that may remain visible in the repository.
It supports:
- structural readiness and posture explanation
- public-safe examples
- community evaluation and improvement
- baseline product differentiation that is already part of SA-NOM's public story

### PT-OSS Advanced

This is the layer that may remain founder-owned and proprietary in the future.
It may include:
- deeper calibration logic
- advanced risk interpretation
- sector or mission-specific structural packs
- premium evaluation profiles
- private scoring or benchmark methods
- quote-specific structural advisory assets

### PT-OSS Founder Judgment

Some value should remain outside code entirely.
This includes:
- founder interpretation
- rollout sign-off
- architecture judgment
- remediation direction
- selective strategic review

Not all premium value needs to become a product module.
Some of it should remain founder-led by design.

## Commercial Boundary

The commercial layer should continue to mean:
- guided evaluation
- paid pilot
- strategic rollout advisory
- selective enterprise, sovereign, or government programs
- partner-supported execution where required

Commercial should not mean:
- retroactively closing the public baseline
- pretending the public product is not real
- forcing customers to pay to unlock what the repository already publicly includes

Commercial can include both:
- delivery and rollout support around the public baseline
- future founder-owned innovation that is intentionally built outside the repository

## Development Rule For New Features

Every new idea should be classified before it is released.

Ask these questions first:
1. Is this part of the honest public baseline?
2. Is this necessary for the public product story to remain credible?
3. Is this a premium differentiator that should stay founder-owned?
4. Is this better treated as founder judgment or commercial delivery instead of public code?

If the answer is "premium differentiator" or "founder-owned innovation," the default path should be:
- do not put it in the public repository yet
- develop it in a private repo, private package, or founder-controlled internal layer
- decide later whether any subset should move into the public baseline

## Public Contribution And Ownership Rule

To protect founder-owned future innovation, SA-NOM should keep this discipline:
- anything accepted into the public repository should be treated as part of the public baseline under the repository license posture
- do not assume future exclusive control over public contributions unless contributor rights are explicitly handled
- do not mix founder-owned premium work into the public repo casually and expect to pull it back later

If contributor volume grows in the future, SA-NOM should consider a clearer contributor rights policy for any area where relicensing or founder-exclusive commercial use may matter.

## Repo And Package Separation Rule

When the time comes to ship future premium innovation, keep the separation clear.

Recommended structure:
- this repository = public baseline
- private internal repository or package = founder-owned advanced modules
- commercial delivery assets = quote-specific materials, private packs, or partner-supported deployment materials

This is cleaner than mixing open and private innovation in one place and trying to explain the split later.

## What This Policy Protects

This policy protects four things at once:
- trust in the public repository
- room for future founder-owned innovation
- a clean open-core posture
- a premium commercial path that does not depend on bait-and-switch behavior

## Short Founder Rule

Use this sentence internally when deciding what to release:

"What is already public stays part of the real baseline. What is truly new, deeper, and premium does not enter the public repo by default."
