# Roadmap - v0.5.2

This roadmap opens a small `v0.5.2` patch line after the governed HITL milestone and the `v0.5.1` dashboard/operator improvements.

## Theme

Tighten the evaluation experience without opening a new concept phase.

## Milestone Goal

By the end of `v0.5.2`, SA-NOM should feel easier to try, easier to inspect, and slightly smoother in the operator-facing runtime surface.

This phase is intentionally narrow.
It should clean up rough edges around dashboard flow, evaluation guidance, and UX friction that appears during real trial use.

## Core Position

`v0.5.2` should behave like a disciplined patch line:
- no new strategic layer
- no new product narrative
- no major structural expansion

Instead, this phase should improve what already exists in `v0.5.0` and `v0.5.1` so the system feels more ready for serious hands-on evaluation.

That means the phase should answer questions such as:
- can a new evaluator understand how to start the private dashboard more quickly
- can an operator follow approval and review lanes with less friction
- can the docs point new users to the right evaluation path without guessing
- can obvious UX gaps or buglets discovered during trial runs be cleaned up before the next larger phase

## Primary Structure

### 1. Dashboard Polish

Target areas:
- small flow improvements in the private dashboard
- clearer operator context in approval, override, Human Ask, and health views
- smoother first-run behavior where small UX confusion still exists

Expected outcomes:
- the operator surface feels more coherent
- evaluation users get stuck less often
- the dashboard looks more deliberate and production-minded without changing the product story

### 2. Evaluation Path Polish

Target areas:
- docs that help a new user choose the right trial path faster
- clearer guidance for local evaluation, private server mode, and dashboard-first review
- better linking between quick-start, guided evaluation, and operator-facing docs

Expected outcomes:
- new evaluators can get to a meaningful demo faster
- fewer owner-guided explanations are needed for basic first-run use
- the evaluation story feels more self-serve

### 3. Patch-Level Bug Fixes And UX Gaps

Target areas:
- small bugs found during trial use
- awkward wording, unclear labels, or confusing control points
- small logic mismatches between dashboard messaging and runtime posture

Expected outcomes:
- fewer rough edges during real usage
- less friction in approval and operator review flows
- a cleaner base for the next major phase

## Non-Goals

- no new enterprise concept layer in `v0.5.2`
- no expansion into the planned global harmony direction yet
- no broad runtime redesign
- no large integration catalog work
- no release scope that reads like a new minor version

## Candidate Deliverables

- small dashboard polish improvements
- evaluation-path and onboarding doc refinements
- patch-level bug fixes from real usage feedback
- `v0.5.2` planning and release materials

## Exit Criteria

- the dashboard feels smoother for first-run operator evaluation
- the docs help new users start the right evaluation path more quickly
- small trial-use friction points are reduced without expanding the product scope
- `v0.5.2` reads as a clean patch release that strengthens the serious-evaluation baseline
