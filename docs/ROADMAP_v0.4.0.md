# Roadmap - v0.4.0

This roadmap opens the `v0.4.0` phase after `v0.3.0` closed the first-run usability milestone.

## Theme

Make PTAG fully public, legible, and contributable as an open governance-language layer without weakening the governed-runtime position of SA-NOM.

## Milestone Goal

By the end of `v0.4.0`, PTAG should read as a public specification and reference implementation that anyone can study, use, evaluate, and improve from the repository itself.

## Core Position

`v0.4.0` is the PTAG public-opening phase.

This milestone is not only a wording refresh. It should align repository language, public documentation, examples, parser and validator references, contribution guidance, and release framing so PTAG is no longer described as a proprietary framework in the public repo.

## Workstreams

### 1. PTAG Public Positioning

Target areas:
- remove outdated proprietary-language claims from public-facing PTAG materials
- define a clear public statement for PTAG availability under the repository license posture
- align README, framework docs, release notes, and announcement language

Expected outcomes:
- public readers no longer see conflicting claims about PTAG openness
- PTAG has a stable public message that can be reused in releases, posts, and conversations
- the repository communicates PTAG as an open governance-language layer, not a half-disclosed internal concept

### 2. PTAG Specification Hardening

Target areas:
- fuller PTAG structure and syntax documentation
- canonical explanation of headers, blocks, validation rules, and semantic expectations
- example-rich explanation that maps public docs to actual parser and validator behavior

Expected outcomes:
- PTAG becomes easier to understand without reading implementation files first
- outside readers can reason about PTAG as a language, not only as product terminology
- the public repo moves closer to a real specification baseline

### 3. PTAG Reference Surface

Target areas:
- clearer surfacing of parser, validator, semantic, and role resource files already in the repo
- public-safe example files for simple and enterprise-shaped PTAG usage
- visible link between PTAG docs and repository artifacts

Expected outcomes:
- PTAG no longer feels partially hidden
- new readers can move from concept to concrete examples faster
- contributors can see where specification and implementation meet

### 4. PTAG Contribution Path

Target areas:
- contributor guidance for PTAG documentation, examples, syntax changes, and validation behavior
- explicit expectations for tests and documentation updates when PTAG behavior changes
- issue-draft framing for future PTAG work

Expected outcomes:
- external contributors have a clear entry path
- PTAG changes become easier to review as a coherent workstream
- the repo is better prepared for public discussion around the language itself

## Non-Goals

- no broad rewrite of the governed execution core just because PTAG is becoming more public
- no public claim that PTAG is already a finished industry standard
- no legal or regulatory overclaim about what PTAG alone guarantees
- no announcement that outruns the repository artifacts actually available to readers

## Candidate Deliverables

- rewritten `docs/PTAG_FRAMEWORK.md` with public-opening language
- updated `README.md` and `CONTRIBUTING.md` for PTAG openness and contribution
- PTAG examples/spec references surfaced more clearly from repository docs
- `v0.4.0` issue drafts, roadmap, release notes, and announcement copy

## Exit Criteria

- the public repo no longer describes PTAG as proprietary
- PTAG docs explain the language structure, validation model, and example usage clearly
- readers can find the PTAG implementation and example artifacts from the documentation path
- contributors can understand how to propose PTAG improvements without guesswork
- `v0.4.0` closes with a coherent public-opening story rather than isolated wording edits
